# Миграции и сиды

Слой данных реализован. PostgreSQL хранит реляционное состояние, а MinIO
хранит байты файлов; оба провижинятся двумя npm-скриптами, которые вы
запускаете **на хосте**, по порядку:

```bash
npm run db:migrate   # 1. создать схему в PostgreSQL
npm run db:seed      # 2. загрузить статические каталоги + создать бакет MinIO
```

Запустите их один раз на свежем стеке и перезапускайте любой в любой момент —
оба идемпотентны (см. [Идемпотентность](#идемпотентность)).

> **Предварительные требования.** PostgreSQL и MinIO должны быть доступны, а
> `.env` должен существовать. Обычная локальная настройка — `cp .env.example .env`,
> затем `docker compose up -d postgres minio` (см.
> [local-development.md](local-development.md)). Оба скрипта читают настройки
> подключения прямо из `.env`, поэтому значения по умолчанию указывают на
> `localhost`.

> **Почему на хосте.** `db:migrate` запускает `drizzle-kit`, а `db:seed`
> работает под `ts-node` — оба являются **dev-зависимостями**. Рантайм-образ
> Docker устанавливает только production-зависимости (`npm ci --omit=dev`),
> поэтому ни одного из этих инструментов нет внутри контейнера бэкенда. Скрипты
> запускаются на хосте и подключаются к опубликованным Compose-ом портам
> `localhost`. CI делает ровно это: его E2E job запускает `db:migrate`, затем
> `db:seed` против PostgreSQL/MinIO в сервис-контейнерах перед тестами (см.
> [ci.md](ci.md)).

## Схема и миграции

Схема Drizzle лежит в `src/infrastructure/database/schema/`, сгруппированная по
функциональным областям (`game-session`, `gameplay`, `commerce`, `presentation`,
`evaluation`, плюс `_shared` enum'ы/колонки). Она компилируется в одну
закоммиченную миграцию:

```
src/infrastructure/database/migrations/0000_wild_enchantress.sql
```

Эта миграция создаёт **16 таблиц** в пяти областях:

| Область | Таблицы |
|---|---|
| `game-session` | `rooms`, `teams`, `players`, `presentation_topics` |
| `gameplay` | `categories`, `questions`, `board_cells` |
| `commerce` | `qr_tools`, `shop_items`, `purchases`, `inventory_items` |
| `presentation` | `presentation_requirements`, `presentation_submissions` |
| `evaluation` | `evaluation_criteria`, `evaluation_scores`, `final_results` |

Скрипты миграций:

| Скрипт | Инструмент | Что делает |
|---|---|---|
| `npm run db:migrate` | `drizzle-kit migrate` | Применить ожидающие миграции к базе данных из `DATABASE_URL`. |
| `npm run db:generate` | `drizzle-kit generate` | Перегенерировать SQL из TypeScript-схемы. **Офлайн** — не подключается. |
| `npm run db:check` | `drizzle-kit check` | Проверить согласованность журнала миграций. |

Когда вы меняете схему, запустите `db:generate` и закоммитьте перегенерированный
SQL. **CI это проверяет:** job quality-gate перезапускает `db:generate` и падает,
если тот выдаёт любой diff под `migrations/` — то есть закоммиченная миграция
разошлась со схемой (см. [ci.md](ci.md)).

## Сиды

`npm run db:seed` запускает `src/infrastructure/database/seeds/seed.ts`. За один
проход он валидирует JSON каталогов, провижинит MinIO, затем делает upsert
каталогов в PostgreSQL:

1. **Валидация** семи JSON-файлов под
   `src/infrastructure/database/seeds/data/`. Валидация двухслойная: пофайловые
   `zod`-схемы фиксируют форму каждой строки и предписанные планом количества, а
   реляционный проход проверяет межфайловые инварианты (нет дублирующихся ID
   между каталогами; каждая категория содержит ровно пять канонических ценовых
   ступеней 100/200/400/600/800 на позициях 0–4; строгое соответствие 1:1
   shop-item ↔ QR-tool без сирот; присутствуют оба обязательных заголовка
   evaluation-criteria). Некорректный или несогласованный файл прерывает работу
   одной агрегированной ошибкой **до любой записи** — никогда не частичный сид.
2. **Провижининг MinIO**: создать бакет, если он отсутствует, применить
   анонимную политику публичного чтения и загрузить один placeholder-QR `.svg` на
   каждый `qr_tool` (чтобы каждый засиженный `storageKey` / `publicUrl` указывал
   на реальный объект). См. [minio.md](minio.md).
3. **Upsert каталогов** в PostgreSQL внутри одной транзакции, упорядоченный так,
   чтобы внешние ключи разрешались (`categories → questions`,
   `qr_tools → shop_items`; topics / requirements / criteria независимы).

Засиженные строки (точные количества, зафиксированные JSON-файлами):

| Каталог (таблица) | Строк |
|---|---|
| `categories` | 6 |
| `questions` | 30 |
| `presentation_topics` | 4 |
| `qr_tools` | 6 |
| `shop_items` | 6 |
| `presentation_requirements` | 4 |
| `evaluation_criteria` | 2 |

Плюс **6 placeholder-QR SVG** (по одному на `qr_tool`), загруженных в MinIO.

Примечания:

- **Семь из шестнадцати таблиц засиживаются** — статические каталоги выше.
  Остальные девять (`rooms`, `teams`, `players`, `board_cells`, `purchases`,
  `inventory_items`, `presentation_submissions`, `evaluation_scores`,
  `final_results`) — **рантайм**-таблицы, создаваемые и заполняемые по ходу
  игры; сид никогда их не трогает.
- **Правильные ответы на вопросы и значения очков — данные только для бэкенда**
  каталога. Они живут в `questions` и никогда не раскрываются клиентам через
  read-модели поля / вопроса.
- **QR SVG — это placeholder'ы**, а не настоящие QR-коды: валидная заглушка с
  рамкой и центрированной подписью, чтобы у пайплайна хранения были реальные
  объекты для отдачи. Подставьте настоящие изображения, заменив ассеты и
  перезапустив сид.

## Идемпотентность

Каждый шаг безопасно перезапускать; ничто не дублируется:

- **Каталоги** делают upsert по своему первичному ключу
  (`ON CONFLICT (id) DO UPDATE`). Второй запуск обновляет производные от JSON
  колонки на месте. `created_at` исключён из набора обновления, поэтому исходная
  временная метка сохраняется.
- **Бакет** создаётся только когда отсутствует (сперва `bucketExists`); политика
  публичного чтения переприменяется каждый запуск (no-op, когда уже установлена).
- **QR SVG** записываются под детерминированными ключами
  (`qr-tools/<qrToolId>.svg`), так что перезапуск перезаписывает тот же объект, а
  не добавляет новый.

Итог: перезапуск `db:seed` обновляет каталоги и ассеты в соответствие с JSON и
никогда не порождает дублирующиеся строки или осиротевшие объекты.

## Порядок и когда запускать

- **Свежий стек:** `db:migrate`, затем `db:seed`. Сид ожидает, что таблицы
  существуют.
- **После подтягивания изменений схемы:** `db:migrate`, чтобы применить новые
  миграции. Перезапустите `db:seed`, если изменился JSON каталогов.
- **После редактирования JSON каталогов:** `db:seed` снова — upsert обновляет
  затронутые строки.
- Успешный сид переводит `storage` в `GET /api/health` в зелёный: бакет теперь
  существует (проба storage проверяет ровно это).

## Устранение неполадок

- **`db:migrate` / `db:seed`: connection refused (ECONNREFUSED)** — PostgreSQL
  или MinIO не подняты, либо `.env` указывает на недоступный адрес. Запустите
  инфраструктуру (`docker compose up -d postgres minio`) и убедитесь, что `.env`
  использует `localhost`.
- **`Missing required environment variable …`** — `.env` отсутствует или
  неполон. Запустите `cp .env.example .env`. Сид читает `DATABASE_URL`,
  `MINIO_*` и им подобные напрямую.
- **Падение из-за расхождения схемы в CI (`db:generate` выдал diff)** — вы
  изменили схему без перегенерации. Запустите `npm run db:generate` локально и
  закоммитьте обновлённый вывод `migrations/`.
- **`Seed data … validation failed` / `relational validation failed`** —
  JSON-файл каталога некорректен или нарушает инвариант (неверное число строк,
  сбитая ценовая ступень, осиротевший shop item / QR tool). Агрегированная ошибка
  перечисляет каждое нарушение; исправьте JSON и перезапустите.
- **Повторный сид после правок не меняет ничего неожиданного** — это upsert
  работает как задумано: существующие строки обновляются по `id`, `created_at`
  сохраняется, и ни одна строка не дублируется.
