# Проект: бэкенд игры «Своя игра: собери презентацию проекта»

NestJS + TypeScript + PostgreSQL + Drizzle + WebSocket + MinIO + Docker.
Scope — только Backend и DevOps. Frontend, UI/UX и дизайн — внешние потребители REST API, WebSocket-событий и публичных файловых ссылок.

## Источник истины
- Канон правил и архитектуры — `.ai/docs/backend_devops_final_plan.md` (финальный план, разделы 1–27). Главный источник.
- `.ai/docs/Этап2_Архитектура_Backend.md` — архитектура: слои, модули, границы, порты/адаптеры.
- `.ai/docs/Этап1_Анализ_и_фиксация_правил.md` — контекст; часть решений устарела (см. §3 Этапа 2).
- Приоритет при конфликте: финальный план → Этап 1 как контекст → явно помеченное предположение.
- Перед любой задачей читай релевантные разделы плана, указанные в промпте. Не реконструируй правила по памяти.

## Жёсткие правила
- НЕ выдумывай модули, поля, эндпоинты, WebSocket-события, бизнес-правила, имена и зависимости, которых нет в плане. Если плану чего-то не хватает — скажи об этом, не достраивай молча.
- Всегда отделяй «так в плане» от «это предложение по реализации».
- Архитектура: Clean + Hexagonal. Слои `domain / application / infrastructure / presentation`; выход во внешний мир — только через порты (интерфейсы) и адаптеры. Domain не знает о NestJS / HTTP / WebSocket / Drizzle / MinIO.
- Этап 3 (базовая инфраструктура) — ОБРАЗЕЦ СТИЛЯ. Следуй его конвенциям: структура модуля, DI-токены портов, именование файлов/классов.
- Минимальные локальные изменения. Никакого постороннего рефакторинга и выхода за scope задачи.
- Открытые значения из раздела 25 плана (длительности таймеров, штраф за просрочку и т.п.) — НЕ хардкодить: выносить в env и подтверждать у оператора.
- Секреты не коммитить.

## Рабочий процесс
- Каждый под-этап — на отдельной ветке `feature/<под-этап>`.
- Готовность доказывается зелёными проверками (typecheck, lint, build, тесты; при наличии — `docker compose up` + health-check), а не на словах. Приводи реальный вывод команд.
- Независимое ревью — в отдельной чистой сессии по diff ветки.

## Отклонения от плана (авторизовано оператором)
- CI: используем **GitHub Actions** вместо GitLab CI (план описывает GitLab). Детали миграции — отдельной задачей.
